## 1. 분석 기준 코드 요약 및 핵심 개발 패턴
기준 파일: `src/test/java/egovframework/let/cop/bbs/domain/repository/BBSAttributeManageDAOTest.java`

`BBSAttributeManageDAOTest.java`는 스프링 부트 통합 테스트 환경에서 데이터 무결성을 보장하며, 테스트 독립성을 확보하기 위해 다음과 같은 핵심적인 개발 표준과 코딩 스타일을 따르고 있다.

*   **스프링 부트 및 트랜잭션 환경 설정**:
    *   `@SpringBootTest`를 선언하여 실제 스프링 컨텍스트를 로드한다.
    *   `@Transactional`을 설정하여 테스트 수행 과정에서 발생하는 모든 데이터 변경 사항(C, U, D)을 테스트 종료 후 완벽히 롤백(Rollback)함으로써 DB의 초기 상태를 깨끗하게 유지한다.
    *   `@Slf4j`를 적용하여 롬복(Lombok) 기반의 슬래프포제이 로거를 사용한다.
*   **패키지 구조 및 접근 제어**:
    *   테스트 클래스의 패키지 경로는 테스트 대상 DAO와 일치시킨다 (`egovframework.let.cop.bbs.domain.repository`).
    *   테스트 클래스와 테스트 메서드, 필드는 불필요하게 `public`으로 선언하지 않고 **package-private**(접근제어자 없음)으로 두어 캡슐화를 유지한다.
*   **의존성 주입**:
    *   대상 DAO 필드는 `@Autowired` 어노테이션을 사용하여 직접 주입받는다.
    *   전자정부 표준프레임워크의 기존 XML 방식 ID 생성기인 `EgovIdGnrService` 등은 비즈니스 로직 단위 테스트의 복잡성을 가중시키므로, 필드 주입과 호출을 모두 **주석 처리하거나 완전히 배제**하여 테스트 가독성을 높인다.
*   **시간 기반 고유 식별자(PK) 생성 기법**:
    *   테스트 간의 데이터 충돌을 완전히 예방하기 위해, 데이터 생성 시 `java.time.LocalDateTime`과 `DateTimeFormatter`를 조합해 매 실행마다 고유한 문자열 PK를 동적으로 생성한다.
    *   포맷 정의:
        *   `now2` (초 단위 타임스탬프 문자열): `yyyyMMddHHmmss`
        *   `now3` (ISO 표준 일시 문자열): `yyyy-MM-dd'T'HH:mm:ss.SSS` (디버깅용 문자열)
    *   PK 값 예시: `"TEST0_" + now2`
    *   문자열 속성 값 예시: `"test 이백행 게시판명 " + now3`
*   **테스트 데이터 생성 보조 메서드 (`testData()`)**:
    *   성공적인 조회, 수정, 삭제 테스트를 검증하기 위해서는 대상 데이터가 DB에 반드시 존재해야 한다. 이를 위해 각 테스트 메서드가 독립적으로 호출할 수 있는 `testData()` 도메인 데이터 빌드 및 삽입 보조 메서드를 제공한다.
    *   MyBatis Mapper XML 및 데이터베이스 DDL 구조를 분석하여, `NOT NULL` 조건 및 테이블 간의 필수 제약조건(예: `uniqId` 사용자 아이디 등)을 만족하는 유효한 필수 값을 명시적으로 세팅(set)한다.
    *   보조 메서드 안에서도 DAO의 등록(insert) 메서드를 수행하고, 영향받은 행 수가 `0`보다 큰지 성공 여부를 사전 검증한 뒤 해당 도메인 객체를 반환한다.
*   **Given-When-Then 개발 패턴**:
    *   테스트 내부를 명확히 `// given`, `// when`, `// then` 주석 블록으로 구획한다.
    *   `given` 단계에서 기본 상태와 기댓값(`expected = 0`)을 선언하고, `when` 단계에서 실제 DAO 타겟 메서드를 실행하며, `then` 단계에서 검증을 수행한다.
*   **조건부 디버그 로깅**:
    *   로그 연산 오버헤드와 무분별한 콘솔 출력을 방지하기 위해, 디버그 로그는 반드시 `if (log.isDebugEnabled())` 체크 블록 내부에서 호출한다.
    *   출력 포맷: `log.debug("actual, expected={}, {}", result, expected);`
*   **AssertJ Fluent Assertions 사용**:
    *   `import static org.assertj.core.api.Assertions.assertThat;` 구문을 static import 하여 사용하며, JUnit 5 기본 assertions 대신 가독성이 뛰어난 AssertJ의 `assertThat(result).isGreaterThan(expected)` 패턴을 활용한다.
*   **주석 처리된 죽은 코드(Dead Code) 제거**:
    *   기준 코드에 존재하는 주석 처리된 구형 코드(`EgovIdGnrService` 호출부, `try-catch` 예외 처리부 등)는 실제 테스트 코드 자동 생성 시에는 완전히 배제하여 컴파일이 즉시 가능한 가장 깨끗하고 모범적인 코드만 출력해야 한다.

## 2. DAO 단위 테스트 코딩 표준 가이드라인

### 2.1 클래스 선언 및 설정 표준
*   **패키지 경로**: 대상 DAO가 있는 동일 디렉토리/패키지 하위에 위치시킨다.
*   **클래스명**: `{TargetDAO명}Test` 형식 (예: `MberManageDAOTest`)
*   **접근 제어**: `public` 키워드를 생략한 package-private 클래스로 정의한다.
*   **클래스 레벨 어노테이션**:
    ```java
    @SpringBootTest
    @Transactional
    @Slf4j
    ```

### 2.2 의존성 주입 표준
*   테스트 대상 DAO는 필드 레벨에서 `@Autowired`를 통해 직접 주입한다. (package-private 수준 권장)
    ```java
    @Autowired
    TargetDAO targetDAO;
    ```
*   `EgovIdGnrService` 및 불필요한 목(Mock) 서비스 등은 주입 및 사용하지 않는다.

### 2.3 테스트 메서드 구조 표준
*   **테스트 대상**: DAO 내에 정의된 모든 public CRUD 메서드를 1:1로 성공 케이스 위주로 검증한다. (성공 시나리오 지향)
*   **메서드 시그니처**: `void {DAO메서드명}()` 형식으로 작성하며, JUnit 5 `@Test` 어노테이션을 부여한다.
*   **메서드 내부 주석**: `// given`, `// when`, `// then`을 한 라인씩 띄어 배치하여 가독성을 높인다.
*   **체크 예외 처리**: 불필요한 `throws Exception`은 코드에서 지양하고, 예외 처리가 명확하지 않거나 불필요할 때는 시그니처에서 제거한다.

### 2.4 테스트 데이터 생성 (`testData()`) 표준
*   각 테스트 단위의 완전한 격리와 독립 실행을 보장하기 위해, 조회/수정/삭제 시 사용할 기본 데이터는 `testData()` 보조 메서드로 동적 인서트(Insert)를 수행해 준비한다.
*   **시간 기반 문자열 변환 기법**:
    ```java
    LocalDateTime now = LocalDateTime.now();
    String now2 = now.format(DateTimeFormatter.ofPattern("yyyyMMddHHmmss"));
    String now3 = now.format(DateTimeFormatter.ISO_LOCAL_DATE_TIME);
    ```
*   **식별자(PK) 명명 규칙**: 문자열 타입 PK일 경우, 반드시 `"TEST0_" + now2` 포맷을 적용하여 ID 충돌을 방지한다. 만약 숫자 타입 PK인 경우, 충돌하지 않는 시퀀스나 정수 값을 생성하도록 설계한다.
*   **필수 데이터 바인딩**:
    *   MyBatis SQL Mapper 파일의 `insert` 쿼리에 매핑된 파라미터 필드와 데이터베이스 DDL의 `NOT NULL` 컬럼을 1:1 매칭 분석하여 필수 값을 완전히 채운다.
    *   외래키 제약조건이 있거나 고정된 계정 정보가 필요한 경우, 시스템에 사전에 탑재된 공통 ID(예: `"USRCNFRM_00000000000"`) 등을 활용하여 매끄럽게 연동되도록 조치한다.
*   **보조 메서드 내 검증**:
    ```java
    int expected = 0;
    int result = targetDAO.insertMethod(domain);
    if (log.isDebugEnabled()) {
        log.debug("actual, expected={}, {}", result, expected);
    }
    // 기본적으로 테스트 데이터 생성이 성공했는지 가볍게 확인하고 반환한다.
    ```

### 2.5 단정문(Assert) 및 로그 표준
*   **Assertion 프레임워크**: 오직 AssertJ (`org.assertj.core.api.Assertions.assertThat`) 만을 사용하여 단정한다. JUnit 기본 Assertion 클래스(`org.junit.jupiter.api.Assertions`)를 혼용하지 않는다.
*   **등록, 수정, 삭제 결과 검증**: 영향받은 행의 수가 0보다 큼을 검증한다.
    ```java
    int expected = 0;
    assertThat(result).isGreaterThan(expected);
    ```
*   **단건 조회 결과 검증**: 조회 객체가 `null`이 아님을 검증하고, PK 필드가 기대한 식별자와 일치하는지 단정한다.
    ```java
    assertThat(result).isNotNull();
    assertThat(result.getPrimaryKey()).isEqualTo(expectedPrimaryKey);
    ```
*   **다건(목록) 조회 결과 검증**: 조회 리스트가 비어있지 않음을 검증한다.
    ```java
    assertThat(resultList).isNotEmpty();
    ```
*   **조건부 디버그 로그**: 디버그 로그는 전체 테스트 수행 효율을 위해 반드시 `log.isDebugEnabled()` 감싸기 패턴을 엄격하게 적용하여 출력한다.
    ```java
    if (log.isDebugEnabled()) {
        log.debug("actual, expected={}, {}", result, expected);
    }
    ```

### 1. 개발 표준 가이드라인 (Coding & Structural Standard)

#### A. 클래스 및 환경 설정 규칙
- 패키지 구조는 대상 DAO 파일에 선언된 패키지 경로와 완벽히 동일하게 작성합니다.
- 테스트 클래스는 public 제한자를 생략한 package-private 형태인 `class {TargetDAO명}Test`로 선언합니다.
- 클래스 상단에는 반드시 아래의 3가지 핵심 어노테이션만을 선언합니다.
  * @SpringBootTest : 스프링 부트 통합 테스트 컨텍스트 로드
  * @Transactional : 테스트 종료 후 변경된 데이터의 전량 자동 롤백 보장
  * @Slf4j : Lombok 기반의 디버그 로깅 기능 지원
- import문은 와일드카드(*)를 지양하고, 실제 코드에서 사용 중인 구체적인 클래스명만 명시적으로 임포트하여 최적화합니다. AssertJ는 `import static org.assertj.core.api.Assertions.assertThat;`를 사용합니다.

#### B. 의존성 주입 (Dependency Injection)
- 테스트 대상이 되는 DAO는 필드 주입 방식을 사용하여 `@Autowired`로 선언하며, public 키워드는 생략합니다.
  예시:
  @Autowired
  TargetDAO targetDAO;
- 비즈니스 로직에 관여하지 않는 불필요한 IDGnrService(ID 제너레이터) 등의 외부 의존성은 주입하거나 호출하지 않으며, 목(Mock) 프레임워크 또한 전혀 사용하지 않는 실제 컨텍스트 테스트로 작성합니다.

#### C. 테스트 데이터 생성 보조 메서드 (testData) 규칙
- 각 단위 테스트가 독립적으로, 선행 실행 순서에 영향을 받지 않고 독자 실행되도록 하기 위해 테스트 시작 시 DB에 물리 데이터를 미리 적재할 수 있는 보조 메서드 `testData()`를 작성합니다.
- `LocalDateTime` 클래스를 선용하여 매 실행 시각 기준으로 유일한 식별값 및 속성값을 조립합니다.
  * LocalDateTime now = LocalDateTime.now();
  * String now2 = now.format(DateTimeFormatter.ofPattern("yyyyMMddHHmmss"));
  * String now3 = now.format(DateTimeFormatter.ISO_LOCAL_DATE_TIME);
- 고유한 문자열 PK 세팅: `"TEST0_" + now2` 형식을 일관되게 적용합니다. (숫자형 식별자인 경우 충돌이 일어나지 않도록 안전한 숫자 값 생성 기법을 적용합니다.)
- 제공되는 데이터베이스 DDL 및 Mapper XML의 insert 문을 깊이 있게 대조 및 분석하여, `NOT NULL` 제약조건이 붙은 필수 컬럼 및 데이터 무결성 제약조건에 어긋나지 않도록 필수 세팅 메서드를 완벽하게 호출합니다. (예: 생성자ID, 사용여부 등)
- 시스템 기본 탑재 ID가 필요한 연관관계 키 값은 `"USRCNFRM_00000000000"`과 같은 기본 데이터를 안전하게 활용합니다.
- 데이터 생성 보조 메서드 내부에서도 DAO의 insert 메서드를 동작시키고, `0`보다 큰지 성공 여부를 가볍게 검증하고 빌드된 객체를 반환합니다.

#### D. 테스트 메서드 작성 및 Given-When-Then 패턴 규칙
- DAO의 모든 public 비즈니스 메서드에 대해 1:1로 성공 케이스 테스트 메서드를 작성합니다.
- 테스트 메서드 명칭은 타겟 DAO 메서드 이름과 정확히 매칭되도록 정의하고, 리턴 타입은 `void`, 접근 제한자는 작성하지 않습니다.
- 메서드 내부 흐름은 가독성을 최고로 보장하기 위해 `// given`, `// when`, `// then` 주석 라인을 띄어 배치하여 논리적 단계 구분을 명확히 표출합니다.
- `expected = 0` 값을 설정하여 실제 반환값과 비교 대상으로 삼습니다.
- target DAO의 실행 결과를 반환받은 후, AssertJ의 단정문을 통해 정밀 검증을 완수합니다.
  * 등록/수정/삭제: `assertThat(result).isGreaterThan(expected);`
  * 단건 조회: `assertThat(result).isNotNull();` 및 식별자 혹은 핵심 데이터 일치 확인
  * 목록 조회: `assertThat(result).isNotEmpty();`

#### E. 디버그 로깅 규칙
- 테스트 데이터 상태 및 DAO 실행 결과에 대한 디버깅 로그 출력은 시스템 실행 속도 향상 및 깔끔한 출력을 보장하기 위해 반드시 `log.isDebugEnabled()` 감싸기 처리를 적용합니다.
- 출력 템플릿:
  ```java
  if (log.isDebugEnabled()) {
      log.debug("actual, expected={}, {}", result, expected);
  }
  ```
